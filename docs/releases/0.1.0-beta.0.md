# Lazuli 0.1.0-beta.0 declaration

- **Declared:** 2026-07-16
- **Channel:** Beta
- **Qualification:** Repository-verified
- **Production authorization:** Not granted
- **Strategy ledger:** `partial=26`, `complete=0`

## What beta means

This release is suitable for controlled evaluation by maintainers and invited testers. The product implementation, migrations, contracts, automated tests, security checks, operational controls, recovery tooling, and guarded acceptance harness are present on the release branch. CI and supply-chain checks passed for commit `30b810418e352f514cf5d0b9bac07a6287028b45` before this declaration.

Beta is a release-maturity label. It does not change the six-condition completion contract in [`PRODUCT-STRATEGY.md`](../../PRODUCT-STRATEGY.md), and it does not convert unexecuted operational procedures into evidence.

## Current operating posture

- Production has not been deployed or changed by this declaration.
- Staging realtime ingestion and publishing remain disabled after validation.
- Retail monetization remains outside the milestone.
- REST polling, modeled liquidation data, proxy CVD, synchronous backtests, and the legacy realtime implementation remain available as documented fallbacks.
- The strategy ledger remains partial until item-specific production evidence exists.

## Deferred beta risks and waivers

The following are accepted beta limitations, not passed gates:

- The 2,000-client 60-minute capacity test has not passed on the final release image.
- Five 2,000-client reconnect cycles and the continuous 72-hour staging soak have not run to acceptance.
- Final live liquidation p95 below 800 ms has not been demonstrated on the final image; the last exact-image 50-client run received no liquidation samples.
- A repeatable custom-domain WebSocket `1006` disconnect was observed in one beta diagnostic, although clients recovered through canonical snapshots without sequence gaps.
- Discord and Telegram staging delivery credentials were intentionally deferred, so A3 cannot receive final provider acceptance.
- Two-profile browser acceptance, failure drills, production rollout windows, and final production SLO observation remain outstanding.

## Promotion boundary

Promotion from Beta to production-verified/GA requires the operational acceptance in [`docs/operations`](../operations/README.md), explicit production approval with a valid change-window ID, and all 26 ledger items satisfying their six evidence conditions. None of those requirements is waived by this declaration.
